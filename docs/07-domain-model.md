# 1. Purpose

## 1.1 Overview

Данный документ описывает предметную область (Domain Model) платформы **GameRent**.

В отличие от `database-design.md`, который описывает структуру хранения данных, данный документ описывает бизнес-сущности системы, их обязанности, взаимосвязи и правила поведения.

Главная цель документа — сформировать единую модель предметной области, независимую от технологий реализации.

Настоящий документ не описывает:

* структуру таблиц PostgreSQL;
* HTTP API;
* внутреннее устройство Go-приложения.

Он отвечает исключительно на вопрос:

> **Как устроен бизнес аренды игровых аккаунтов?**

---

# 2. Domain Vision

GameRent предоставляет пользователям возможность временного доступа к игровым аккаунтам Steam без передачи прав собственности на аккаунт.

Основной бизнес-процесс системы:

1. Пользователь проходит регистрацию.
2. Пользователь выбирает игровой аккаунт.
3. Пользователь оплачивает аренду.
4. Система предоставляет временный доступ к аккаунту.
5. По окончании аренды доступ прекращается.
6. История аренды сохраняется.
7. На основании истории формируется уровень доверия пользователя.

Таким образом, ключевой объект предметной области — **аренда (Rental)**, а не пользователь и не игровой аккаунт.

* публикацию аккаунтов;
* доступность аккаунтов.

Основная сущность:

* Account

---

## 4.3 Rentals Context

Центральный контекст всей системы.

Отвечает за:

* создание аренды;
* жизненный цикл аренды;
* продление аренды;
* завершение аренды;
* отмену аренды.

Основная сущность:

* Rental

## 4.5 Games Context

Основная сущность:

* Game

---

## 4.6 Security Context

Отвечает за:

# 5. Aggregates

Предметная область построена вокруг четырёх основных агрегатов.

## 5.1 Rental (Aggregate Root)

Rental является центральным агрегатом системы.

Он управляет:

* временем аренды;
* пользователем;
* игровым аккаунтом;
* стоимостью;
* депозитом;
* статусом аренды.

Ни одна бизнес-операция не может изменить аренду, минуя Aggregate Root.

---

## 5.2 Account

Account представляет игровой аккаунт Steam.

Он отвечает за:

* библиотеку игр;
* публикацию;
* стоимость;
* доступность;
* состояние аккаунта.

Account не отвечает за оплату или управление пользователями.

---

## 5.3 User

User представляет зарегистрированного пользователя платформы.

User управляет:

* профилем;
* историей аренд;
* уровнем доверия;
* отзывами;
* уведомлениями.

User не управляет игровыми аккаунтами напрямую.

---

## 5.4 Payment

Payment является отдельным агрегатом.

Он отвечает за:

* проведение оплаты;
* возврат средств;
* удержание депозита;
* фиксацию финансового результата операции.

Payment не изменяет состояние Rental самостоятельно — изменение происходит через бизнес-логику приложения.

---

# 6. Entities

## 6.1 User

User представляет зарегистрированного пользователя платформы.

Основные характеристики:

* уникальный идентификатор;
* адрес электронной почты;
* пароль;
* уровень доверия;
* история аренд.

Жизненный цикл User не зависит от существования аренд.

---

## 6.2 Account

Account представляет Steam-аккаунт.

Основные характеристики:

* SteamID64;
* логин;
* зашифрованный пароль;
* стоимость аренды;
* размер депозита;
* библиотека игр;
* статус публикации.

Account может существовать без активной аренды.

---

## 6.3 Rental

Rental представляет временное соглашение между платформой и пользователем о предоставлении доступа к конкретному Steam-аккаунту.

Rental всегда создаётся для:

* одного пользователя;
* одного аккаунта;
* одного периода времени.

---

## 6.4 Payment

Payment представляет финансовую операцию.

Операция может относиться к:

* оплате аренды;
* возврату;
* удержанию депозита.

---

## 6.5 Game

Game представляет игру Steam.

Game существует независимо от конкретного аккаунта.

Несколько аккаунтов могут содержать одну и ту же игру.

---

## 6.6 Review

Review представляет отзыв пользователя после завершения аренды.

Отзыв связан одновременно с:

* пользователем;
* аккаунтом;
* арендой.

---

# 7. Value Objects

В отличие от Entity, Value Object не обладает собственной идентичностью.

Он определяется исключительно своим значением.

## 7.1 Money

Money представляет денежную сумму.

Содержит:

* Amount
* Currency

Money является неизменяемым объектом.

---

## 7.2 RentalPeriod

RentalPeriod представляет период аренды.

Содержит:

* Start Time
* End Time

Предоставляет операции:

* Duration()
* IsExpired()
* RemainingTime()

---

## 7.3 TrustLevel

TrustLevel определяет уровень доверия пользователя.

Используется при:

* определении размера депозита;
* ограничении доступа;
* анализе истории пользователя.

---

## 7.4 SteamCredentials

SteamCredentials представляет данные доступа к Steam-аккаунту.

Содержит:

* Login
* Encrypted Password
* SteamID64

Пароль никогда не хранится в открытом виде.

---

## 7.5 IPAddress

IPAddress используется для фиксации действий пользователя.

Используется системой безопасности для:

* аудита;
* анализа подозрительной активности;
* уведомления владельца аккаунта при подтверждении входа через Steam Guard.

---

# 8. Relationships

Предметная область содержит следующие основные связи.

* Один User может иметь множество Rental.
* Один Account может участвовать во множестве Rental (в разное время).
* Один Rental относится только к одному User.
* Один Rental относится только к одному Account.
* Один Rental может иметь одну или несколько связанных Payment (например, основная оплата и возврат депозита).
* Один Account содержит множество Game.
* Одна Game может принадлежать множеству Account.
* Один Rental может завершиться созданием одного Review.

---

# 9. General Principles

При проектировании доменной модели используются следующие принципы:

* бизнес-логика не зависит от инфраструктуры;
* сущности описывают поведение, а не только данные;
* агрегаты защищают целостность предметной области;
* каждый бизнес-объект имеет одну область ответственности;
* все ключевые бизнес-правила будут описаны как инварианты и машины состояний в следующих разделах документа.

# 10. Business Rules

Данный раздел описывает правила предметной области, которые должны соблюдаться независимо от реализации приложения, используемой базы данных или пользовательского интерфейса.

---

## 10.1 User Registration

При регистрации пользователь обязан:

* предоставить уникальный адрес электронной почты;
* задать пароль, удовлетворяющий требованиям безопасности;
* принять Пользовательское соглашение;
* принять Политику конфиденциальности.

До подтверждения адреса электронной почты пользователь считается неподтверждённым и не может создавать аренды.

---

## 10.2 Account Publication

Игровой аккаунт может быть опубликован только после прохождения всех проверок.

Минимальные требования:

* аккаунт принадлежит платформе;
* Steam Guard включён;
* инвентарь проверен;
* отсутствуют ценные игровые предметы;
* библиотека игр успешно синхронизирована;
* аккаунт не имеет активной аренды;
* аккаунт не находится в состоянии обслуживания.

Если хотя бы одно условие нарушено, публикация невозможна.

---

## 10.3 Rental Creation

Создание аренды допускается только при выполнении всех условий.

Условия:

* пользователь авторизован;
* аккаунт доступен;
* период аренды допустим;
* стоимость успешно рассчитана;
* пользователь соответствует требованиям Trust Level (если применимо);
* внесён обязательный депозит (если требуется).

После создания аренды аккаунт становится недоступным для новых бронирований.

---

## 10.4 Payment

Перед активацией аренды оплата должна быть успешно завершена.

До подтверждения оплаты доступ к аккаунту не предоставляется.

---

## 10.5 Rental Activation

После успешной оплаты:

* аренда становится активной;
* пользователь получает доступ к данным аккаунта;
* владельцу (администратору платформы) фиксируется факт начала аренды;
* начинается отсчёт времени аренды.

---

## 10.6 Rental Completion

После окончания аренды система:

* завершает аренду;
* прекращает доступ пользователя;
* фиксирует время окончания;
* освобождает аккаунт;
* инициирует возврат депозита (если отсутствуют нарушения).

---

## 10.7 Reviews

Отзыв может быть оставлен только после завершения аренды.

Пользователь может оставить только один отзыв для одной аренды.

Изменение уже опубликованного отзыва не допускается.

---

# 11. State Machines

Для всех основных агрегатов определены допустимые состояния и переходы между ними.

---

## 11.1 Rental State Machine

```
CREATED
    │
    ▼
WAITING_PAYMENT
    │
    ├──────────────► CANCELLED
    │
    ├──────────────► EXPIRED
    │
    ▼
ACTIVE
    │
    ▼
EXPIRED
    │
    ▼
COMPLETED
```

Текущий backend создаёт аренду сразу в `WAITING_PAYMENT`; `CREATED` и `COMPLETED` остаются доменными статусами, но не являются основным API happy-path для создания/закрытия аренды.

### Состояния

**CREATED**

Аренда создана, но ещё не готова к оплате.

---

**WAITING_PAYMENT**

Ожидается успешная оплата.

Account находится в статусе `RESERVED`, payment находится в статусе `PENDING`.

---

**ACTIVE**

Пользователь получил доступ к аккаунту.

Доступ к credentials разрешён только владельцу этой активной, оплаченной и неистёкшей аренды через `GET /api/v1/me/rentals/{rentalId}/credentials`.

---

**EXPIRED**

Оплаченное время закончилось.

Выполняются завершающие операции.

Credentials после перехода в `EXPIRED` не выдаются.

---

**COMPLETED**

Аренда полностью завершена.

В текущем API/worker flow переход в `COMPLETED` не является реализованным закрытием оплаченной аренды; expiration использует `EXPIRED`.

---

**CANCELLED**

Аренда отменена до завершения оплаты.

Реализованный cancel flow разрешён для `WAITING_PAYMENT` и не освобождает account из `ACTIVE`.

---

### Реализованные переходы Rental

| From | To | Trigger | Side effects |
| --- | --- | --- | --- |
| `WAITING_PAYMENT` | `ACTIVE` | successful payment webhook | `payment PENDING -> SUCCESS`, `account RESERVED -> RENTED` |
| `WAITING_PAYMENT` | `CANCELLED` | owner cancel endpoint | `payment PENDING -> FAILED`, `account RESERVED -> AVAILABLE` |
| `WAITING_PAYMENT` | `EXPIRED` | waiting-payment cleanup | `payment PENDING -> FAILED`, `account RESERVED -> AVAILABLE` |
| `ACTIVE` | `EXPIRED` | expired rental cleanup | `payment SUCCESS` сохраняется, `account RENTED -> AVAILABLE` |

Повторные cancel/expire операции идемпотентны: состояние не меняется повторно и audit/security event не дублируется.

---

### Недопустимые переходы

Запрещено:

ACTIVE → WAITING_PAYMENT

ACTIVE → CANCELLED через реализованный cancel endpoint

COMPLETED → ACTIVE

CANCELLED → ACTIVE

COMPLETED → CREATED

---

## 11.2 Account State Machine

```
CREATED
    │
    ▼
VERIFYING
    │
    ▼
AVAILABLE
    │
    ▼
RESERVED
    │
    ▼
RENTED
    │
    ▼
AVAILABLE

AVAILABLE
    │
    ▼
MAINTENANCE
    │
    ▼
AVAILABLE

AVAILABLE
    │
    ▼
DISABLED
```

### Реализованные переходы Account

| From | To | Trigger |
| --- | --- | --- |
| `AVAILABLE` | `RESERVED` | `POST /rentals` создал неоплаченную аренду |
| `RESERVED` | `RENTED` | successful payment webhook активировал аренду |
| `RESERVED` | `AVAILABLE` | cancel `WAITING_PAYMENT` или waiting-payment cleanup |
| `RENTED` | `AVAILABLE` | expiration `ACTIVE -> EXPIRED` |
| any allowed admin state | `MAINTENANCE` / `DISABLED` | admin status update or account ban flow |

Account не должен становиться `AVAILABLE`, если для него существует валидная `ACTIVE` аренда. Для конкурирующих аренд база содержит partial unique index на `account_id` для rental statuses `WAITING_PAYMENT` и `ACTIVE`.

### CREATED

Аккаунт добавлен в систему.

---

### VERIFYING

Проводится проверка:

* Steam Guard;
* инвентаря;
* библиотеки игр.

---

### AVAILABLE

Аккаунт доступен для аренды.

---

### RESERVED

Создана аренда, ожидается завершение оплаты.

---

### RENTED

Аккаунт находится в активной аренде.

---

### MAINTENANCE

Проводится обслуживание аккаунта.

Публикация запрещена.

---

### DISABLED

Аккаунт выведен из эксплуатации.

---

## 11.3 Payment State Machine

```
PENDING
    │
    ├──────────────► FAILED
    │
    ▼
SUCCESS
```

В текущем backend создание rental создаёт payment сразу в `PENDING`. Статусы refund/deposit ledger не реализованы.

---

### CREATED

Создана запись о платеже.

---

### PENDING

Платёж ожидает подтверждения от provider/webhook.

---

### SUCCESS

Оплата подтверждена.

---

### FAILED

Оплата завершилась ошибкой.

---

### Реализованные переходы Payment

| From | To | Trigger | Notes |
| --- | --- | --- | --- |
| `PENDING` | `SUCCESS` | successful webhook | требует `external_transaction_id`; активирует rental |
| `PENDING` | `FAILED` | cancel `WAITING_PAYMENT` | terminal для отменённой неоплаченной аренды |
| `PENDING` | `FAILED` | waiting-payment cleanup | terminal для просроченной неоплаченной аренды |
| `SUCCESS` | `SUCCESS` | repeated webhook with same `external_transaction_id` | идемпотентный no-op |

`provider` и `external_transaction_id` используются для идемпотентности webhook. Отдельный refund/deposit ledger пока не реализован.

# 12. Domain Invariants

Инварианты — это правила, которые никогда не должны нарушаться.

---

## User

Пользователь всегда имеет уникальный email.

---

Пользователь не может иметь отрицательный Trust Level.

---

Удаление пользователя не должно удалять историю его аренд.

---

## Account

Аккаунт может иметь только одну активную аренду одновременно.

В текущей схеме это усилено partial unique index: для одного `account_id` одновременно допускается не более одной rental со статусом `WAITING_PAYMENT` или `ACTIVE`.

---

Аккаунт обязан иметь уникальный SteamID64.

---

Аккаунт не публикуется без успешной проверки безопасности.

---

Аккаунт не публикуется без синхронизированной библиотеки игр.

---

Аккаунт не публикуется при наличии ценных предметов в инвентаре.

---

Пароль аккаунта никогда не хранится в открытом виде.

---

Steam Guard должен быть включён.

---

## Rental

Каждая аренда принадлежит одному пользователю.

---

Каждая аренда относится к одному аккаунту.

---

Дата окончания всегда больше даты начала.

---

Стоимость аренды всегда положительна.

---

Продолжительность аренды должна находиться в допустимых пределах, определённых бизнес-правилами.

---

Активная аренда не может существовать без успешной оплаты.

Credentials не возвращаются из `POST /rentals` или `POST /payments/webhook`; они выдаются только владельцу через `GET /api/v1/me/rentals/{rentalId}/credentials` и только пока rental остаётся `ACTIVE`.

---

После завершения аренды пользователь обязан потерять доступ к аккаунту.

---

## Payment

Сумма платежа всегда больше нуля.

---

Валюта платежа обязательна.

---

Возврат не может превышать сумму первоначального платежа.

Refund/deposit ledger пока не реализован в backend; это правило остаётся целевым ограничением для будущего ledger, а не текущим исполняемым flow.

---

## Review

Отзыв может быть оставлен только после завершённой аренды.

---

Одна аренда допускает только один отзыв.

# 15. Domain Events

## 15.1 Purpose

Domain Event представляет собой факт, который уже произошёл в системе и имеет значение для бизнеса.

Доменные события описывают предметную область и не зависят от конкретной реализации.

На текущем этапе они являются логической моделью.

В MVP не предполагается использование Event Bus или Message Broker, однако события должны учитываться при проектировании сервисов.

---

## 15.2 User Events

### UserRegistered

Создаётся после успешной регистрации пользователя.

Используется для:

* отправки письма;

* создания первоначального Trust Level.

Создаётся после подтверждения электронной почты.

Создаётся при изменении уровня доверия пользователя.

Используется для:

* аналитики.

# 16. Aggregate Interaction

Агрегаты взаимодействуют друг с другом только через публичные сервисы.

Прямое изменение состояния одного агрегата другим запрещено.

Например:

```text
RentalService

↓

AccountRepository

↓

Account
```

а не

```text
Rental

↓

меняет Payment самостоятельно
```

Каждый агрегат отвечает только за собственную целостность.

---

# 17. Aggregate Responsibilities

## User

Отвечает за:

* профиль;
* уровень доверия;
* историю аренд.

Не отвечает за:

* оплату;
* игровые аккаунты;
* библиотеку Steam.

---

## Account

Отвечает за:

* публикацию;
* стоимость;
* доступность;
* библиотеку игр.

Не отвечает за:

* платежи;
* пользователей.

---

## Rental

Отвечает за:

* период аренды;
* пользователя;
* аккаунт;
* стоимость;
* депозит.

Не отвечает за:

* отправку email;
* обновление Steam;
* запись Audit Log.

---

## Payment

Отвечает исключительно за финансовую операцию.

Не управляет жизненным циклом аренды.

---

# 18. Consistency Rules

Система придерживается модели сильной согласованности (Strong Consistency) для критически важных операций.

К ним относятся:

* создание аренды;
* изменение статуса аккаунта;
* проведение оплаты;
* удержание депозита.

Все эти операции должны выполняться в рамках транзакции PostgreSQL.

Некритичные процессы могут выполняться асинхронно.

Например:

* отправка email;
* создание уведомлений;
* аналитика;
* обновление кэша Redis.

---

# 19. Domain Model and Database

Доменные сущности не являются прямым отображением таблиц PostgreSQL.

Например:

```
Rental
```

может использовать данные из нескольких таблиц:

* rentals;
* payments;
* users;
* accounts.

Аналогично:

```
Account
```

может агрегировать информацию из:

* accounts;
* account_games;
* games.

Таким образом, доменная модель описывает бизнес, а не структуру хранения данных.

---

# 20. Domain Model and API

REST API не должен раскрывать внутреннюю структуру доменной модели.

Для взаимодействия используются DTO (Data Transfer Objects).

Это позволяет:

* изменять внутреннюю модель без нарушения совместимости API;
* скрывать чувствительные данные;
* формировать удобные ответы для клиента.

---

# 21. Domain Model and Architecture

Domain Layer располагается между Application Layer и Infrastructure Layer.

```text
HTTP

↓

Handlers

↓

Application Services

↓

Domain Model

↓

Repository Interfaces

↓

Infrastructure

↓

PostgreSQL
```

Бизнес-правила должны находиться в Domain Layer или Application Services.

Repository не должен содержать бизнес-логику.

Handler не должен принимать бизнес-решения.

---

# 22. Go Implementation Guidelines

Каждый агрегат рекомендуется реализовать отдельным пакетом.

Пример:

```
internal/

    rental/

        entity.go

        service.go

        repository.go

        dto.go

        errors.go

        validator.go

        mapper.go
```

Каждый пакет инкапсулирует собственную бизнес-логику.

Взаимодействие между пакетами осуществляется через интерфейсы.

# 23. Future Evolution

Текущая модель поддерживает дальнейшее развитие без нарушения существующих принципов.

Возможные расширения:

* поддержка нескольких игровых платформ;
* система промокодов;
* подписка на платформу;
* система достижений пользователей;
* список избранных аккаунтов;
* рекомендации аккаунтов;
* динамическое ценообразование;
* антифрод-анализ на основе поведения пользователей;
* интеграция с внешними платёжными системами.

Расширение должно происходить через добавление новых агрегатов, сервисов или доменных политик без нарушения существующих инвариантов.

# 24. Conclusion

Доменная модель является центральным элементом архитектуры GameRent.

Она определяет:

* бизнес-сущности;
* их ответственность;
* допустимые состояния;
* правила взаимодействия;
* неизменяемые инварианты;
* жизненный цикл объектов.

Все последующие архитектурные решения — структура базы данных, REST API, сервисный слой и реализация на Go — должны соответствовать настоящей доменной модели.

Изменение бизнес-логики начинается с изменения данного документа. Только после обновления доменной модели допускается изменение архитектуры, базы данных или программного кода.
