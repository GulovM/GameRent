# Software Requirements Specification

# 1. Введение

## 1.1 Назначение

Настоящий документ определяет функциональные и нефункциональные требования к системе аренды игровых аккаунтов GameRent.

Документ является основным источником требований для проектирования архитектуры, базы данных, API и реализации программного обеспечения.

---

## 1.2 Область применения

Система предназначена для автоматизации процесса аренды игровых аккаунтов различных игровых платформ.

Пользователь получает возможность:

- просматривать каталог;
- выбирать игровой аккаунт;
- оплачивать аренду;
- получать временный доступ к игровому аккаунту.

Администратор управляет всеми игровыми аккаунтами, пользователями и системой в целом.

---

## 1.3 Термины

| Термин | Определение |
|---------|-------------|
| Пользователь | Зарегистрированный клиент системы |
| Администратор | Пользователь с административными правами |
| Игровой аккаунт | Аккаунт Steam, Epic Games или другой платформы |
| Аренда | Ограниченный по времени период использования игрового аккаунта |
| Баланс | Внутренний денежный счет пользователя |
| Транзакция | Любое изменение баланса пользователя |
| Сессия | Авторизованное взаимодействие пользователя с системой |

## FR-016. Synchronization of Steam Library

```text
The system shall synchronize the game library of every registered Steam account using the official Steam Web API.

Synchronization shall be performed:
- when the account is first registered;
- manually by an administrator;
- periodically by a background synchronization service.

The synchronized data shall include:
- Steam AppID;
- game title;
- playtime (if available);
- game icon/header image (if available).
```

---

## FR-017. Game Catalog (Поиск)

```text
The system shall maintain a local catalog of all games available on registered Steam accounts.

The catalog shall allow:

- searching accounts by game;
- filtering accounts by owned games;
- displaying game metadata;
- displaying account libraries without querying Steam in real time.
```

---

## FR-018. Background Processing (Добавить новый сервис.)

### Steam Synchronization Worker

```text
A background worker shall periodically synchronize Steam account libraries.

Responsibilities:

- update owned games;
- add newly purchased games;
- update playtime information;
- synchronize game metadata.
```

---

## FR-019. External Systems

Добавить.

```text
Steam Web API

Purpose:

- retrieve owned games;
- retrieve game metadata;
- synchronize account libraries.
```

# Я бы еще немного изменил архитектуру

Пока мы обсуждали безопасность, незаметно родилась очень хорошая архитектура. Я бы ее уже зафиксировал в SRS.

Получается примерно так:

```text
                Client
                   │
                   ▼
              REST API
                   │
 ┌─────────────────┼──────────────────┐
 │                 │                  │
 ▼                 ▼                  ▼
Rental        Account          Catalog
Service       Service          Service
 │                 │                  │
 └────────────┬────┴───────┬──────────┘
              ▼            ▼
          PostgreSQL     Redis

              ▲
              │
      Steam Sync Worker
              │
              ▼
         Steam Web API
```

Обрати внимание, что здесь уже начинают просматриваться **границы сервисов** внутри твоего Go-приложения. Даже если ты реализуешь всё как монолит, логическое разделение на модули (`RentalService`, `AccountService`, `CatalogService`, `SteamSyncWorker`) значительно упростит разработку и последующее сопровождение.

---

## Но есть еще одна идея

Я бы **вообще перестал хранить название игры как основное поле**.

Почему?

Потому что Steam идентифицирует игры через **AppID**.

Например:

```
730  -> Counter-Strike 2
570  -> Dota 2
620  -> Portal 2
578080 -> PUBG
```

Значит:

**Источник истины** — `appid`.

Название, иконка, дата выхода, описание — это **метаданные**, которые можно обновлять отдельно.

То есть таблица `games` должна выглядеть примерно так:

```sql
games
------
id
steam_app_id (UNIQUE)
name
header_image_url
capsule_image_url
release_date
created_at
updated_at
```

А связь:

```sql
account_games
-------------
account_id
game_id
playtime_minutes
last_synced_at
```

# 5. Нефункциональные требования

## Производительность

- время ответа API не более 300 мс при отсутствии высокой нагрузки;
- поддержка конкурентных запросов;
- эффективное использование Redis.

---

## Надежность

- отсутствие двойной аренды;
- использование транзакций PostgreSQL;
- восстановление после сбоев.

---

## Безопасность

- JWT Authentication;
- Refresh Tokens;
- RBAC;
- bcrypt;
- защита от SQL Injection;
- защита от XSS;
- защита от CSRF (при необходимости);
- Rate Limiting.

---

## Масштабируемость

Архитектура должна позволять:

- горизонтальное масштабирование API;
- добавление новых игровых платформ;
- расширение бизнес-логики без изменения существующих компонентов.

---

## Поддерживаемость

Система должна быть построена с использованием принципов Clean Architecture.

# 6. Бизнес-правила

BR-001

Один игровой аккаунт может иметь только одну активную аренду одновременно.

---

BR-002

Игровой аккаунт должен быть недоступен для новой аренды до завершения текущей аренды.

---

BR-003

Стоимость аренды определяется установленным тарифом игрового аккаунта.

---

BR-004

После окончания аренды игровой аккаунт автоматически становится доступным.

---

BR-005

При недостаточном балансе аренда невозможна.

---

BR-006

Удаление игрового аккаунта с активной арендой запрещено.

---

BR-007

Пользователь получает учетные данные игрового аккаунта только во время активной аренды.

---

BR-008

После завершения аренды доступ к учетным данным прекращается.

---

BR-009

Все финансовые операции выполняются атомарно.

---

BR-010

Все действия администратора журналируются.

---

# 7. Ограничения

На этапе MVP:

- все игровые аккаунты принадлежат сервису;
- реальные платежные системы отсутствуют;
- отсутствует маркетплейс между пользователями;
- отсутствует мобильный клиент.
