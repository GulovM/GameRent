# Development Roadmap — GameRent

**Проект:** GameRent  
**Язык:** Go 1.25+  
**Архитектура:** Clean Architecture + Domain-Driven Design (DDD)  
**Статус:** Базовая спецификация MVP  

Настоящий документ определяет пошаговый план разработки и развёртывания платформы аренды игровых аккаунтов GameRent. План разбит на логические этапы (спринты), сгруппированные по технологическим и бизнес-доменам на основе требований SRS, архитектурного регламента и модели предметной области.

---

## Этап 1: Инфраструктурный фундамент и общее окружение (Недели 1–3)
**Цель:** Создание структуры проекта, настройка базовых библиотек, конфигурации, логирования и механизмов миграции базы данных.

### 1.1. Инициализация репозитория и окружения
- Настройка базовой структуры каталогов в соответствии с архитектурным регламентом: `cmd/`, `internal/`, `pkg/`, `api/`, `configs/`, `deployments/`, `migrations/`.
- Настройка `Makefile` с базовыми командами (`make build`, `make test`, `make lint`, `make migrate`).
- Конфигурация статического анализа: настройка правил `golangci-lint`, `gofmt` и `goimports`.

### 1.2. Общая инфраструктура (`internal/shared`)
- **Config (`internal/shared/config`):** Интеграция библиотеки `Viper`. Реализация загрузки конфигурации из YAML-файлов и переменных окружения с разграничением уровней (Local, Dev, Staging, Production). Секреты (JWT, ключи шифрования) настраиваются строго через ENV.
- **Logger (`internal/shared/logger`):** Настройка структурированного логирования с использованием `Uber Zap`. Реализация обязательных полей контекста: `request_id`, `user_id`, `ip_address`, `method`, `path`, `latency`, `status_code`.
- **Database (`internal/shared/database`):** Интеграция пула соединений `pgxpool` для PostgreSQL. Реализация интерфейса `Transaction Manager` для управления границами транзакций на уровне сервисов.
- **Cache (`internal/shared/cache`):** Настройка клиента Redis для последующего кэширования некритичных данных (карточки игр, результаты поиска).
- **Clock (`internal/shared/clock`):** Реализация интерфейса `Clock` для изоляции системного времени и обеспечения тестируемости временных инвариантов.

### 1.3. Базовые Middleware и обработка ошибок
- Реализация единой модели ошибок (`internal/shared/errors`) с категориями: Validation, Unauthorized, Forbidden, NotFound, Conflict, Internal.
- Разработка стандартного формата ответов API (`internal/shared/response`).
- Создание сквозных Middleware: `Request ID`, `Recovery`, `Logging`, `Timeout` (по умолчанию) и `Rate Limiter` (Token Bucket) для защиты чувствительных эндпоинтов.

## Этап 2: Контекст аутентификации и управление пользователями (Недели 4–5)
**Цель:** Реализация регистрации, авторизации, ротации токенов и базового профиля пользователя.

### 2.1. Модуль Аутентификации (`internal/auth`)
- Разработка сущности `User` и структуры хранения в таблице `users`. Реализация хэширования паролей с помощью `bcrypt`/`Argon2id`.
- Реализация эндпоинтов:
  - `POST /auth/register` — регистрация с созданием записи и отправкой токена подтверждения.
  - `POST /auth/login` — аутентификация с выдачей пары Access Token (15 мин) и Refresh Token (30 дней).
  - `POST /auth/logout` — отзыв токенов и инвалидация сессии.
  - `POST /auth/refresh` — механизм ротации токенов (Refresh Token Rotation) с хэшированием в таблице `refresh_tokens`.
- Создание `Auth Middleware` для проверки JWT и RBAC (роли `USER`, `ADMIN`).

### 2.2. Модуль Пользователей (`internal/user`)
- Разработка эндпоинтов управления профилем: `GET /auth/me`, обновление личных данных.
- Интеграция сервиса уведомлений (`internal/shared/mail`) для отправки писем подтверждения Email.
- Инициализация базового поля `trust_score` (значение по умолчанию, проверка CHECK-ограничения на уровне БД от 0 до 1000).

## Этап 3: Каталог игр и интеграция со Steam (Недели 6–8)
**Цель:** Интеграция со Steam Web API, создание воркера синхронизации и запуск каталога аккаунтов.

### 3.1. Модуль Игр (`internal/game`)
- Проектирование и создание таблиц `games` (на базе `steam_app_id` как источника истины) и `account_games` (хранение `playtime_minutes`).
- Реализация публичных эндпоинтов каталога: `GET /games`, `GET /games/{gameId}` с поддержкой пагинации (`LIMIT`/`OFFSET`), поиска и фильтрации.
- Интеграция кэширования Redis для популярных игр и карточек каталога.

### 3.2. Модуль Аккаунтов (`internal/account`)
- Создание сущности `Account` и таблицы `accounts`. Реализация шифрования учётных данных Steam (`encrypted_password` типа `BYTEA` с алгоритмом AES-256).
- Реализация конечной точки для администраторов: `POST /admin/accounts` для добавления новых Steam-аккаунтов.
- Внедрение машины состояний аккаунта (`CREATED` -> `VERIFYING` -> `AVAILABLE` -> `RESERVED` -> `RENTED` -> `MAINTENANCE` -> `DISABLED`).
- Разработка фонового процесса `Steam Synchronization Worker` (`internal/shared/scheduler`): автоматическое получение библиотек игр, обновление метаданных через Steam Web API и инвалидация кэша в Redis.

## Этап 4: Движок аренды и транзакционная логика (Недели 9–11)
**Цель:** Реализация центрального бизнес-процесса системы — жизненного цикла аренды с контролем состояния гонки и фоновым планировщиком.

### 4.1. Создание аренды и управление транзакциями (`internal/rental`)
- Разработка сущности `Rental` (Aggregate Root) и таблицы `rentals`.
- Внедрение машины состояний аренды (`CREATED` -> `WAITING_PAYMENT` -> `ACTIVE` -> `EXPIRED` -> `COMPLETED` -> `CANCELLED`).
- Реализация эндпоинта `POST /rentals` (Бизнес-процесс **Create Rental**):
  - Открытие транзакции PostgreSQL.
  - Блокировка строки аккаунта через `SELECT FOR UPDATE` для предотвращения двойной аренды.
  - Проверка инвариантов (статус аккаунта `AVAILABLE`, баланс пользователя).
  - Перевод аккаунта в `RESERVED`, создание записи аренды и платежа.
  - Коммит транзакции, возврат ссылки на оплату.

### 4.2. Модуль Платежей и Активация аренды (`internal/payment`)
- Разработка таблицы `payments` с типами платежей (`Rental Payment`, `Deposit Hold`, `Deposit Release`, `Refund`).
- Реализация симулированного вебхука платёжной системы `POST /payments/webhook` (Бизнес-процесс **Rental Activation**):
  - Атомарное обновление статуса платежа в `Success`.
  - Генерация внутреннего доменного события `PaymentSucceeded`.
  - Изменение статуса аренды на `Active` (`2`) и статуса Steam-аккаунта на `Rented` (`4`).
  - Логирование события безопасности `Rental Started`.
  - Предоставление клиенту доступа к расшифрованным данным `SteamCredentials` на оплаченный период.

### 4.3. Фоновый планировщик завершения аренд
- Разработка воркера `CleanupExpiredRentals` в рамках `Background Scheduler`.
- Реализация периодического сканирования таблицы `rentals` на предмет активных аренд, где `end_at` меньше текущего времени по `Clock`.
- Автоматический запуск процесса **Rental Completion**: перевод аренды в `Completed`/`Expired`, освобождение аккаунта в `Available`, инвалидация сессионных токенов доступа и генерация события `RentalCompleted`.

## Этап 5: Безопасность, пересчёт Trust Score и отзывы (Недели 12–13)
**Цель:** Внедрение механизмов антифрода, логирования событий безопасности, расчёта рейтинга надёжности и системы отзывов.

### 5.1. Модуль Безопасности (`internal/security`)
- Реализация таблицы `security_events` для иммутабельного логирования (отключение Soft Delete).
- Разработка `SecurityService` для фиксации типов событий: `Login Attempt`, `Steam Guard Request`, `Rental Started`, `Rental Finished`, `Suspicious Activity`, `Security Incident`.
- Реализация сбора метаданных контекста запроса: `ip_address`, `user_agent`, JSONB `metadata`, интеграция с сервисом геолокации.
- Реализация триггеров алармов для администраторов при фиксации критических инцидентов.

### 5.2. Возврат депозита и Trust Score Recalculation
- Реализация бизнес-процесса **Deposit Release / Return**: асинхронный перехват события `RentalCompleted`, сканирование истории инцидентов за период аренды, автоматическое проведение возврата залога при отсутствии нарушений.
- Разработка `TrustService`: метод `RecalculateTrust`. Асинхронный расчёт числового значения `trust_score` (0–1000) на основании истории успешных аренд и штрафов. Автоматическое обновление уровня доверия (`TrustLevel`: Bronze, Silver, Gold, Diamond) и очистка кэша профиля в Redis.

### 5.3. Модуль Отзывов (`internal/review`)
- Создание таблицы `reviews` с CHECK-ограничением на оценку (1–5).
- Реализация эндпоинта `POST /reviews` (Бизнес-процесс **Submit Review**). Обеспечение инварианта «один отзыв на одну аренду» за счёт уникального частичного индекса `UNIQUE(rental_id)` на уровне СУБД.
- Реализация асинхронного пересчёта среднего рейтинга Steam-аккаунта в PostgreSQL с инвалидацией кэша карточки в Redis.

## Этап 6: Тестирование, стабилизация и развёртывание (Недели 14–15)
**Цель:** Покрытие кода тестами, обеспечение Graceful Shutdown, настройка CI/CD конвейера и финальный релиз MVP.

### 6.1. Тестирование (Testing Strategy)
- Написание Unit-тестов для проверки доменных сущностей, валидаторов и Use Cases сервисного слоя с использованием mock-реализаций интерфейсов репозиториев.
- Разработка интеграционных тестов для проверки работы репозиториев, транзакционных границ PostgreSQL и изоляции уровней в изолированных Docker-контейнерах.
- Проведение E2E-тестирования критических пользовательских сценариев (регистрация -> аренда -> оплата -> завершение).

### 6.2. Стабилизация и производство (Operations)
- Внедрение механизма **Graceful Shutdown** в `main.go`: корректная остановка HTTP-сервера, завершение активных воркеров планировщика, закрытие пула соединений PostgreSQL (`pgxpool`) и Redis, синхронизация логов `Zap`.
- Настройка конфигурации `Docker Compose` для Production (контейнеры `app`, `postgres`, `redis`). Исключение публикации портов СУБД наружу.
- Развёртывание Reverse Proxy (`Nginx`/`Caddy`) перед приложением с терминацией TLS (HTTPS) и настройкой сжатия.
- Настройка конвейера `GitHub Actions` (CI/CD): автоматический запуск линтеров, тестов, сборка многоэтапного (Multi-stage) Docker-образа и деплой по схеме `Rolling Update`.

## Этап 7: Пост-MVP развитие (План на будущее)
- Интеграция с реальными внешними шлюзами платёжных систем ( Stripe, ЮKassa и др.).
- Внедрение полноценной событийно-ориентированной архитектуры (EDA) с брокером сообщений (`Kafka`, `RabbitMQ` или `NATS`).
- Добавление поддержки других игровых платформ (Epic Games Store, EA App, Ubisoft Connect).
- Разработка административной панели на базе gRPC для внутреннего взаимодействия.
- Развёртывание системы мониторинга `Prometheus` и визуализации метрик в `Grafana` на основе эндпоинта `/metrics`.
