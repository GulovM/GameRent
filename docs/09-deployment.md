# Deployment

**Project:** GameRent

**Version:** 1.0

**Status:** Approved

## P1.1 migration rollout note

Apply `20260713150000_rental_completion_deposit_settlement.sql` before deploying the Phase 1 application. The migration gives only consistent historical paid `EXPIRED + HELD` rentals a fresh 24-hour rollout review window; it does not derive historical deadlines from old `end_at`, so deployment cannot immediately mass-release legacy deposits. Preflight should count paid expired holds that remain without a deadline because they are inconsistent and require manual investigation.

Phase 1 exposes the settlement finalizer as an explicit backend job function only. Do not add or enable a production auto-release scheduler loop until the later API/notification rollout supplies an explicit configuration gate and operational monitoring. Rollback must first ensure no application instance depends on the new columns; the Goose Down migration removes the P1.1 fields and indexes.

# 2. Deployment Strategy

GameRent проектируется по принципу **Build Once, Deploy Anywhere**.

Один и тот же Docker-образ используется во всех окружениях.

Изменяется только конфигурация.

Это гарантирует, что код, протестированный в тестовой среде, идентичен коду, работающему в production.

* нагрузочного тестирования;

Боевое окружение.

Особенности:

# 4. Deployment Architecture

Для MVP используется единый сервер.

```text id="k8a2wp"
                Internet
                    │
                    ▼
              Reverse Proxy
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
    GameRent API          Static Files
        │
        ├─────────────┐
        ▼             ▼
   PostgreSQL       Redis
```

Все внутренние сервисы работают в изолированной Docker-сети.

Доступ извне имеют только Reverse Proxy и HTTP API.

# 5. Docker Compose Topology

На первом этапе используются следующие контейнеры.

```text id="h4em0o"
docker-compose

├── app
├── postgres
├── redis
└── mailhog (optional)
```

---

## app

Основной контейнер приложения.

Содержит:

* HTTP API;
* бизнес-логику;
* планировщик фоновых задач;
* интеграции со Steam;
* обработчики уведомлений.

Один контейнер выполняет все функции системы.

---

## postgres

Основная база данных.

Используется официальный Docker-образ PostgreSQL.

Том (Volume) обязательно подключается для сохранения данных.

---

## redis

Используется для:

* кэширования;
* хранения временных данных;
* ограничения частоты запросов (Rate Limiting);
* хранения краткоживущих значений.

Redis не является источником истины.

---

## mailhog

Используется только в локальной разработке.

Позволяет просматривать отправленные письма без использования реального SMTP-сервера.

В production отсутствует.

1. Загрузка конфигурации.
2. Проверка обязательных переменных окружения.
3. Инициализация логгера.
4. Подключение PostgreSQL.

Если любой обязательный этап завершается ошибкой, приложение завершает работу с ненулевым кодом выхода.

# 13. Secrets Management

Все чувствительные данные считаются секретами и никогда не должны храниться в репозитории.

К секретам относятся:

* JWT Secret;
* ключ шифрования Steam-учётных данных;
* пароль PostgreSQL;
* пароль Redis (если используется);
* SMTP-пароль;
* API-ключи внешних сервисов.

---

## Storage

Для MVP секреты передаются через переменные окружения.

В дальнейшем рекомендуется использовать специализированное хранилище секретов, например:

* Docker Secrets;
* HashiCorp Vault;
* AWS Secrets Manager;
* Azure Key Vault.

Ключ шифрования Steam-учётных данных должен существовать отдельно от базы данных.

---

# 14. TLS

В production весь внешний трафик должен использовать HTTPS.

TLS завершается на Reverse Proxy.

Внутри Docker-сети допускается использование HTTP.

---

# 15. Reverse Proxy

Перед приложением размещается Reverse Proxy (например, Nginx или Caddy).

Он отвечает за:

* завершение TLS;
* перенаправление HTTP → HTTPS;
* сжатие ответов;
* ограничение размера запросов;
* проксирование запросов к приложению;
* отдачу статических файлов (при необходимости).

Приложение GameRent не должно самостоятельно заниматься обслуживанием TLS-сертификатов.

---

# 16. Database Migrations

Миграции выполняются автоматически при выпуске новой версии приложения либо отдельным шагом CI/CD.

Основные правила:

* каждая миграция атомарна;
* уже применённые миграции не изменяются;
* миграции применяются последовательно;
* при ошибке процесс развёртывания останавливается.

Перед запуском миграций рекомендуется создавать резервную копию базы данных.

### Preflight для `deposit_holds.status`

Перед применением миграции `20260713120000_deposit_hold_status_check.sql` в production необходимо проверить существующие данные:

```sql
SELECT id, rental_id, status
FROM deposit_holds
WHERE status NOT IN (1, 2, 3, 4);
```

Ожидаемый результат — ноль строк. Если запрос возвращает строки, миграцию нельзя принудительно применять: сначала нужно расследовать причину и выполнить отдельную документированную data-fix процедуру с сохранением audit trail ручных исправлений. Ошибка миграции в этом случае является намеренным fail-safe поведением: неизвестный статус депозита не должен незаметно отображаться как `NONE` (отсутствие депозита).

# 19. Logging

Приложение использует структурированное логирование с помощью Zap.

Логи разделяются по уровням:

* DEBUG;
* INFO;
* WARN;
* ERROR;
* FATAL.

Каждая запись содержит:

* время;
* уровень;
* Request ID;
* IP клиента (если применимо);
* идентификатор пользователя (если аутентифицирован);
* сообщение;
* дополнительные поля.

Логи приложения выводятся в `stdout`/`stderr`, что соответствует рекомендациям контейнеризированных приложений.

# 22. Backup Strategy

Резервное копирование выполняется автоматически.

## PostgreSQL

Ежедневно:

* полный backup.

Периодически:

* архивирование WAL.

---

## Redis

Регулярно сохраняются снимки состояния (RDB Snapshot).

Поскольку Redis используется только как кэш, потеря его данных не приводит к потере бизнес-информации.

---

## Backup Storage

Резервные копии должны храниться:

* минимум в двух экземплярах;
* на отдельном носителе или в отдельном хранилище;
* с ограниченным доступом.

---

# 23. Restore Procedure

При восстановлении выполняются следующие шаги.

1. Остановка приложения.
2. Восстановление PostgreSQL.
3. Восстановление Redis (при необходимости).
4. Запуск миграций (если требуется).
5. Запуск приложения.
6. Проверка Health Endpoint.
7. Проверка целостности данных.

Восстановление должно регулярно тестироваться.

---

# 24. Horizontal Scaling

Архитектура допускает горизонтальное масштабирование приложения.

```text id="em7d0r"
             Load Balancer
                   │
      ┌────────────┼────────────┐
      ▼            ▼            ▼
   API #1       API #2       API #3
           │
           ▼
      PostgreSQL

           │
           ▼
          Redis
```

Все экземпляры приложения являются stateless (не хранят состояние пользователя в памяти процесса).

Это позволяет масштабировать сервис без изменения бизнес-логики.

---

# 25. Rolling Update

Обновление production должно выполняться по схеме Rolling Update.

Последовательность:

1. Запуск нового экземпляра.
2. Проверка готовности (`/health/ready`).
3. Переключение части трафика.
4. Остановка старого экземпляра.

Это минимизирует простой системы.

---

# 26. Disaster Recovery

Для восстановления после серьёзных сбоев необходимо иметь:

* актуальные резервные копии базы данных;
* резервные копии конфигурации;
* резервные копии секретов;
* Docker Compose и Dockerfile;
* журнал применённых миграций.

Документация по восстановлению должна храниться отдельно от исходного кода.

---

# 27. Security Recommendations

Production-сервер должен соответствовать следующим требованиям.

* Доступ по SSH только по ключам.
* Запрещён вход под пользователем `root`.
* Используется межсетевой экран (Firewall).
* Регулярно устанавливаются обновления безопасности.
* Docker работает от отдельного пользователя.
* PostgreSQL и Redis недоступны из внешней сети.
* Все секреты имеют минимально необходимые права доступа.
* Все административные действия журналируются.

---

# 28. Deployment Checklist

Перед каждым релизом необходимо убедиться, что:

* все тесты успешно пройдены;
* линтер не обнаружил ошибок;
* Docker-образ успешно собран;
* миграции успешно применяются;
* резервная копия базы данных создана;
* секреты обновлены (при необходимости);
* `/health/live` возвращает успешный статус;
* `/health/ready` возвращает успешный статус;
* мониторинг и логирование работают корректно.

# 30. Conclusion

Стратегия развёртывания GameRent ориентирована на простую эксплуатацию, безопасность и постепенное масштабирование.

Для MVP используется минимальный набор компонентов — приложение, PostgreSQL и Redis в Docker Compose. При этом архитектура остаётся готовой к дальнейшему росту без изменения бизнес-логики.

Основными принципами являются:

* единый Docker-образ для всех окружений;
* конфигурация через переменные окружения;
* автоматизированные миграции;
* централизованное логирование;
* мониторинг состояния системы;
* регулярное резервное копирование;
* безопасное хранение секретов;
* готовность к горизонтальному масштабированию.
