# Security Architecture — GameRent

## 1. Цель документа

Данный документ описывает модель угроз, принципы безопасности и архитектуру защиты системы GameRent — платформы аренды игровых аккаунтов Steam.

Система работает в условиях внешней неконтролируемой платформы (Steam), поэтому не может гарантировать абсолютную защиту аккаунтов. Вместо этого она реализует многоуровневую модель снижения риска, контроля доступа и аудита.

---

## 2. Модель угроз (Threat Model)

Система рассматривает следующие классы угроз:

### 2.1. Компрометация игрового аккаунта

- смена пароля
- смена email
- включение Steam Guard на стороннее устройство
- попытка закрепления аккаунта за пользователем

---

### 2.2. Мошенническое поведение арендатора

- использование читов в играх (VAC/game bans)
- попытки несанкционированного изменения данных аккаунта
- злоупотребление функциями Steam (trade, market, gifting)

---

### 2.3. Повторное использование доступа

- сохранение сессий
- повторный вход после завершения аренды
- использование украденных credentials вне системы

---

### 2.4. Атаки на платформу GameRent

- brute force атак на аккаунты пользователей
- SQL injection / API abuse
- компрометация административных интерфейсов
- утечка базы данных

---

### 2.5. Недобросовестные администраторы

- несанкционированный доступ к учётным данным
- изменение данных аренды
- извлечение информации о аккаунтах

---

## 3. Ограничения системы безопасности

Система НЕ может гарантировать:

- отсутствие VAC / game bans
- отсутствие игровых блокировок со стороны Steam
- отсутствие действий пользователя вне клиента GameRent
- контроль поведения внутри Steam клиента

Все риски, связанные с политиками Steam, считаются внешними и неустранимыми.

---

## 4. Принципы безопасности

### 4.1. Least Privilege

Каждый компонент системы имеет минимально необходимый уровень доступа.

---

### 4.2. No Direct Credential Exposure

Логин и пароль Steam не отображаются пользователю в UI системы.

---

### 4.3. Defense in Depth

Безопасность реализуется на нескольких уровнях:

- до аренды
- во время аренды
- после аренды
- финансовый уровень
- поведенческий уровень

---

### 4.4. Auditability

Все критические действия логируются:

- аренда
- входы
- IP
- устройство
- изменения статуса
- финансовые операции

---

## 5. Архитектура безопасности

### 5.1. Credential Storage

Steam credentials:

- хранятся в PostgreSQL
- зашифрованы AES-256
- ключ шифрования хранится отдельно (ENV/Vault)

---

### 5.2. Access Control (RBAC)

Роли:

- USER
- ADMIN

Доступ строго ограничен на уровне API.

---

### 5.3. Authentication

- JWT Access Token
- Refresh Token rotation
- session invalidation on logout

---

## 6. Lifecycle безопасности аренды

### 6.1. До аренды

Система проверяет:

- Trust Level пользователя
- статус аккаунта
- наличие активной аренды
- риск-профиль пользователя

---

### 6.2. Во время аренды

Система фиксирует:

- IP пользователя
- User-Agent
- географию (approx)
- устройство (если доступно)
- длительность сессии
- активность API

---

### 6.3. После аренды

- аренда закрывается автоматически
- доступ к credentials уничтожается
- сессия пользователя инвалидируется
- создаётся audit log

---

## 7. Trust Level System

### 7.1. Назначение

Trust Level определяет доступ пользователя к аккаунтам разной ценовой категории и уровню риска.

---

### 7.2. Уровни

- Bronze
- Silver
- Gold
- Diamond

---

### 7.3. Факторы расчёта

- история аренд
- количество успешных завершений
- наличие жалоб
- подозрительная активность
- стабильность IP / устройства

---

## 8. Security Deposit (страховой депозит)

### 8.1. Назначение

Финансовый механизм покрытия потенциального ущерба.

---

### 8.2. Применение

Депозит может быть удержан в случае:

- нарушения правил использования аккаунта
- получения игрового бана по вине пользователя
- попыток несанкционированного доступа

---

### 8.3. Условия применения

Решение об удержании основывается на:

- audit logs
- risk analysis
- правилах платформы Steam
- внутреннем расследовании

---

## 9. Event Correlation Layer

Система сопоставляет события:

- попытки входа Steam
- начало аренды
- IP адреса
- временные метки

Цель — выявление:

- несоответствий
- подозрительных входов
- аномалий поведения

---

## 10. Notification System (Owner Awareness Layer)

Владельцы аккаунтов получают:

### 10.1. Информационные уведомления

- начало аренды
- активная сессия
- завершение аренды

### 10.2. Security notifications

- попытка входа в аккаунт
- географическое несоответствие
- подозрительная активность

⚠ Эти уведомления НЕ являются механизмом подтверждения доступа.

---

## 11. Data Protection

### 11.1. User data

- хранение персональных данных минимизировано
- логируется только необходимая информация для безопасности

---

### 11.2. Logging

Логи включают:

- IP
- device fingerprint (если доступен)
- timestamps
- actions
- session metadata

---

## 12. Incident Handling

### 12.1. Типы инцидентов

- подозрение на взлом аккаунта
- подозрение на читинг
- попытка обхода аренды
- финансовые аномалии

---

### 12.2. Реакция системы

- блокировка аренды
- заморозка аккаунта
- увеличение Trust penalty
- уведомление администратора

---

## 13. Финальная модель безопасности

Система реализует не абсолютную защиту, а:

> risk-managed controlled access system

Основные механизмы:

- Trust Level (поведенческая модель)
- Security Deposit (финансовый риск)
- Event Correlation (аналитика событий)
- Encryption layer (защита данных)
- Audit system (прозрачность)
- Access lifecycle control (контроль аренды)

---

## 14. Итоговое ограничение

GameRent не контролирует Steam и не может предотвращать все возможные нарушения внутри платформы.

Система обеспечивает:

- контроль доступа к аккаунтам
- снижение риска злоупотреблений
- отслеживание и аудит действий
- финансовое покрытие части рисков
