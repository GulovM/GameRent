# 1. Purpose

## 1.1 Goal

Настоящий документ описывает последовательности взаимодействия компонентов системы при выполнении основных пользовательских и внутренних сценариев.

Диаграммы отражают:

* порядок выполнения операций;
* взаимодействие между слоями приложения;
* работу с PostgreSQL и Redis;
* интеграцию со Steam;
* работу фоновых процессов;
* обработку ошибок.

Все диаграммы соответствуют архитектуре, описанной в `architecture.md`.

# 2. Registration

## Description

Пользователь создаёт новую учётную запись.

```text id="4r5jnt"
User
 │
 ▼
HTTP Handler
 │
 ▼
Auth Service
 │
 ├────────► Validator
 │
 ├────────► User Repository
 │
 │             │
 │             ▼
 │        PostgreSQL
 │
 ├────────► Mail Service
 │
 ▼
HTTP Response
```

### Flow

1. Пользователь отправляет форму регистрации.
2. Выполняется валидация входных данных.
3. Проверяется уникальность email.
4. Пароль хэшируется.
5. Пользователь сохраняется в PostgreSQL.
6. Генерируется токен подтверждения Email.
7. Отправляется письмо.
8. Клиент получает успешный ответ.

### Sequence Diagrams: Part 2 — Core Business Processes

### 13. Rental Activation

**Description** Процесс перевода аренды в активное состояние после успешного подтверждения транзакции со стороны платёжной системы. Операция является критической и выполняется в единой транзакции базы данных для обеспечения строгой согласованности.

**Flow** 1. Платёжная система отправляет HTTP-вебхук с подтверждением успешной транзакции на эндпоинт `POST /payments/webhook`.
2. `Payment Handler` перехватывает запрос, валидирует подпись и передаёт управление в `Payment Service`.
3. `Payment Service` запускает транзакцию PostgreSQL через общий `Transaction Manager`.
4. Статус записи в таблице `payments` изменяется на `2` (`Success`).
5. `Payment Service` генерирует внутреннее событие `PaymentSucceeded`.
6. `Rental Service` перехватывает событие внутри транзакции и обновляет статус аренды в таблице `rentals` на `2` (`Active`).
7. `Account Service` обновляет состояние арендуемого Steam-аккаунта в таблице `accounts` с `3` (`Reserved`) на `4` (`Rented`).
8. `Security Service` регистрирует новое событие безопасности (`2` — `Rental Started`) с сохранением `IPAddress` и `User-Agent` клиента.
9. `Transaction Manager` фиксирует (commit) транзакцию в PostgreSQL.
10. `Notification Service` асинхронно отправляет информационное письмо владельцу аккаунта о старте сессии аренды, а клиент получает доступ к расшифрованным данным `SteamCredentials`.

---

### 14. Rental Completion

**Description** Сценарий завершения аренды по инициативе пользователя или при автоматическом истечении времени. Включает отзыв доступа и освобождение ресурсов.

**Flow** 1. Клиент отправляет запрос на эндпоинт `POST /rentals/{rentalId}/cancel` (или триггер срабатывает из фонового планировщика `Scheduler`).
2. `Rental Handler` вызывает Use Case метод в `Rental Service`.
3. `Rental Service` открывает транзакцию PostgreSQL.
4. Статус аренды в таблице `rentals` меняется на `4` (`Completed`), фиксируется время в поле `actual_finished_at`.
5. `Account Service` переводит статус Steam-аккаунта из состояния `4` (`Rented`) в состояние `2` (`Available`) для последующих бронирований.
6. Сессионные токены доступа для данного периода аренды инвалидируются, закрывая клиенту легитимный доступ к отображению `SteamCredentials`.
7. `Security Service` делает запись в таблицу `security_events` со статусом `3` (`Rental Finished`).
8. `Audit Log` фиксирует системное изменение агрегата. Транзакция закрывается.
9. `Rental Service` порождает доменное событие `RentalCompleted`, которое инициирует асинхронную цепочку проверки и возврата страхового депозита.

### 15. Deposit Release / Return

**Description** Автоматический возврат денежного залога (Deposit) на баланс пользователя после закрытия аренды, если со стороны системы безопасности не было зафиксировано нарушений.

**Flow** 1. Асинхронный обработчик событий перехватывает доменное событие `RentalCompleted` и вызывает `TrustService` совместно с `SecurityService`.
2. `SecurityService` сканирует таблицу `security_events` на наличие критических инцидентов (`4` — `Suspicious Activity`, `6` — `Security Incident`) за время прошедшего периода аренды.
3. При отсутствии зарегистрированных инцидентов и жалоб `TrustService` одобряет возврат депозита.
4. `Payment Service` инициирует транзакцию возврата средств.
5. В таблице `payments` создаётся запись с типом `2` (`Deposit Release`) и статусом `0` (`Created`).
6. Система отправляет запрос во внешний шлюз платёжной системы (или переводит средства на внутренний `Balance` пользователя в MVP).
7. После успешного перевода статус платежа обновляется до `2` (`Success`).
8. `Notification Service` отправляет пользователю уведомление типа `3` (`Deposit Released`).

### 16. Submit Review

**Description** Публикация отзыва и выставление оценки аккаунту после успешного окончания аренды. База данных жёстко контролирует инвариант «один отзыв на одну аренду».

**Flow** 1. Клиент отправляет POST-запрос на `/reviews` с указанием `rental_id`, `rating` (от 1 до 5) и текстового комментария.
2. `Review Handler` пропускает данные через `validator.go` (проверка обязательных полей и диапазона оценки).
3. `Review Service` проверяет по базе данных, что аренда имеет статус `Completed` и данный `user_id` действительно являлся её арендатором.
4. Выполняется попытка записи в таблицу `reviews`. Благодаря ограничению `UNIQUE(rental_id)` на уровне СУБД, повторный отзыв блокируется.
5. После успешной вставки `Review Service` отправляет задачу в воркер на пересчёт кэша.
6. Происходит пересчёт среднего рейтинга Steam-аккаунта в PostgreSQL, и обновлённая карточка аккаунта инвалидируется/перезаписывается в кэше Redis.

### 17. Security Event Logging

**Description** Сквозной процесс логирования любых чувствительных или потенциально опасных действий в системе. Записи носят иммутабельный (неизменяемый) характер.

**Flow** 1. Внутренний компонент приложения (например, `Auth Middleware` при неудачном входе или `AccountVerificationService` при сбое валидации Steam Guard) вызывает `SecurityService`.
2. `SecurityService` считывает метаданные контекста запроса: `ip_address`, `user_agent`, а также через сервис геолокации определяет страну и город подключения.
3. Формируется JSONB-объект `metadata`, содержащий специфичные для конкретного инцидента технические параметры (например, количество неудавшихся попыток, изменённые поля).
4. `Security Service` осуществляет прямую запись в таблицу `security_events` без возможности последующего обновления или мягкого удаления (Soft Delete для данной таблицы отключен).
5. Если тип события классифицируется как `4` (`Suspicious Activity`) или `6` (`Security Incident`), мгновенно генерируется событие для `Notification Service` для отправки `Security Alert` администраторам платформы.

### 18. Trust Score Recalculation

**Description** Асинхронный пересчёт рейтинга надёжности пользователя (`trust_score`) на основании его поведения и истории завершённых аренд.

**Flow** 1. События `RentalCompleted` или `SecurityIncidentRegistered` служат триггером для запуска метода `RecalculateTrust` в `TrustService`.
2. `TrustService` запрашивает из PostgreSQL агрегированную информацию по пользователю: общее число успешных аренд, частоту смены IP-адресов, наличие штрафов за удержание аккаунта.
3. Применяется внутренняя доменная политика (Domain Policy) для расчёта нового целочисленного значения в диапазоне от 0 до 1000.
4. `TrustService` обновляет поле `trust_score` в таблице `users`. На уровне БД срабатывает `CHECK`-ограничение на валидность диапазона.
5. Если новый рейтинг пересекает установленные бизнес-метриками пороговые значения, изменяется системный уровень доверия пользователя (`TrustLevel`: Bronze, Silver, Gold, Diamond).
6. Происходит принудительная очистка кэша профиля данного пользователя в Redis.

### 19. Background Scheduler (Expired Rentals Cleanup)

**Description** Периодический фоновый процесс, отслеживающий и принудительно завершающий те аренды, чей оплаченный временной интервал подошёл к концу.

**Flow** 1. Фоновый планировщик `Scheduler` (пакет `internal/shared/scheduler`), работающий по таймеру, инициирует задачу `CleanupExpiredRentals`.
2. Воркер делает запрос в базу данных PostgreSQL, выбирая все записи из таблицы `rentals`, где `status = 2` (`Active`) и текущее системное время (полученное через интерфейс `Clock`) строго больше значения `end_at`.
3. Для каждой найденной просроченной аренды воркер вызывает метод принудительного закрытия в `RentalService`.
4. `RentalService` инициирует стандартный процесс завершения аренды (см. разд. 14): переводит состояние записи в `3` (`Expired`), освобождает Steam-аккаунт в `Available`, фиксирует аудит-логи.
5. Инициируется запуск сессии `Steam Guard` для сброса активных подключений на устройстве арендатора (если поддерживается интеграцией).
6. `Notification Service` отправляет пользователю уведомление типа `2` (`Rental Completed / Expired`) с сообщением о прекращении доступа.
